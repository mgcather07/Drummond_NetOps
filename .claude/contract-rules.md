# Contract Rules

How a project's **system contracts** are kept honest: schemas, API
endpoints, and system docs that other code depends on live in one
place, carry a version and a date, and can be **locked** so they
cannot drift without the user signing off. **Read this file when a
task touches anything under `contracts/`, or when wiring up the
contract guard.**

The `/contract` skill implements everything below.

## What a contract is

A **contract** is a definition other code is bound to — change it
carelessly and something breaks far from the edit. Three kinds:

- **`schema`** — a database schema, a data model, a message shape.
- **`endpoint`** — an API endpoint's request/response contract.
- **`doc`** — a system-level document other work depends on.

Each contract is a stamp under `contracts/stamps/<name>.md`: YAML
frontmatter (the machine-readable identity — see `stamps.md`) plus
a body (the actual definition). Each is versioned (`vMAJOR.MINOR.PATCH`)
and date-stamped (`created`, `last_updated`).

## The folder

```
contracts/
├── CONTRACTS.md      index rollup — regenerated by /contract
├── LEDGER.md         append-only audit of every change
└── stamps/
    └── <name>.md     one contract stamp per file
```

`contracts/` sits at the repo root, alongside `tasks/`, `env/`, and
`tests/` — first-class, project-owned. It is created on demand by
`/contract init`, not by `bin/init`.

## The cardinal rule

**Nothing under `contracts/` is hand-edited. Ever.** Every change
goes through `/contract`, which writes the stamp, stamps the date,
and records a ledger entry in the same operation. A change that
skips the skill skips the ledger — and a ledger with holes is worse
than no ledger, because it looks trustworthy and isn't.

This is enforced, not just asked:

- A **`PreToolUse` guard hook** denies the `Edit`, `Write`, and
  `MultiEdit` tools on any path under `contracts/`. It installs into
  `.claude/settings.json` (committed, shared) so every contributor's
  session is held to it.
- The **`contract.sh` script** is the only sanctioned way to mutate
  a contract, and it refuses a locked one outright.

The guard does not police raw shell — a `Bash` redirection into
`contracts/` is out of its reach. It is not adversary-proof; it is
honest-mistake-proof, which is the realistic failure mode. The
script-level lock check is the hard guarantee for the sanctioned
path.

## The lock

A contract's frontmatter carries `is_locked: true | false`.

**A locked contract cannot change.** `contract.sh update` and
`contract.sh bump` refuse a locked contract with exit code 3, and
the guard hook denies the edit. This is deliberate friction.

**When a task needs to change a locked contract:**

1. **Stop the task.** Do not work around the lock — not by
   unlocking it, not by raw shell, not by editing a copy.
2. **Surface it to the user.** Name the contract, state the change
   the task needs, and quote why it's locked (the lock's ledger
   entry).
3. **Wait.** The user decides: unlock it (`/contract unlock <name>
   --why "…"`) and let the task proceed, or stop the task.

Unlocking is **always** the user's call. An agent that quietly
unlocks a contract to get unblocked has defeated the entire
mechanism. The lock exists precisely so that a change to a load-
bearing definition is a conscious, recorded decision.

Before a task starts work that might touch a contract, it can
pre-flight with `contract.sh check <path>` — exit 3 means locked.

## The ledger

`contracts/LEDGER.md` is an **append-only** record. Every contract
change — created, updated, version-bumped, locked, unlocked — adds
an entry:

- **When** — timestamp, written by the script.
- **Who** — `git config user.name` (the actor making the change).
- **What** — the concrete change.
- **Why** — the reason, supplied by the user or the task as
  `--why "<reason>"`. Required on every mutating verb. Plain fact,
  not narrative: "Frozen ahead of the v3 backend cutover" — not
  "Locked to ensure stability and alignment."

The ledger is itself under `contracts/` and therefore also guarded
— it is written only by `contract.sh`, never by hand.

## For tasks

- A task that changes a schema, an endpoint shape, or a system doc
  is changing a contract. Route the change through `/contract`, not
  through a direct edit.
- If the contract is locked, the task is **blocked** until the user
  resolves it (see "The lock"). Treat this like any other hard
  blocker: document it in the task's blocker notes and stop.
- Adding a *new* contract is not blocked — `new` is always allowed.
  Locking applies to *changing an existing* contract.

## Relationship to other kit machinery

- **`stamps.md`** — defines the `contract` stamp model. The
  frontmatter shape lives there.
- **`/schema-check`** — reconciles a *local mirror* against an
  *externally-owned* schema. `/contract` manages the schemas this
  project *owns*. Different jobs: one watches what you copy, the
  other governs what you publish.
- **`/install-hook`** — `/contract` uses it to install and remove
  the guard hook.
- **Cross-repo contract sharing** — out of scope for this file.
  Linking one repo's `contracts/` to another's, and scanning for
  drift between them, is a separate, later feature.
